# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Bilingual (Arabic RTL / Turkish LTR) medical clinic appointment-booking system. Two fully decoupled halves that only talk over REST at `/api/v1`:

- `backend/` — FastAPI + SQLAlchemy. DB-agnostic (SQLite dev, PostgreSQL prod).
- `frontend/` — Static HTML/CSS/JS. **No build step, no framework, no bundler.**

The decoupling is intentional: a mobile app could replace the frontend without touching the backend.

## Commands

**Backend** (from `backend/`):
```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env                 # then edit SECRET_KEY, admin creds
python -m app.seed                   # create tables + admin (+ demo data unless SEED_DEMO_DATA=false)
uvicorn app.main:app --reload        # http://127.0.0.1:8000  (docs at /docs)
```

**Frontend** (from `frontend/`): any static server, e.g. `python3 -m http.server 5500`. The backend's `CORS_ORIGINS` must list the exact origin serving the frontend.

**Docker** (from `backend/`): `docker compose up -d --build`. Two services only: `clinic-api` + `cloudflared` — the DB is **external managed Postgres (Neon)**, there is no db container. Required stack env vars: `SECRET_KEY`, `DATABASE_URL` (Neon string rewritten to SQLAlchemy form `postgresql+psycopg2://…?sslmode=require`, using the pooled `-pooler` host), `FIRST_ADMIN_PASSWORD`, `TUNNEL_TOKEN`. The entrypoint runs `python -m app.seed` on every container start (idempotent). `SEED_DEMO_DATA` defaults to `false` in compose (fresh prod DB starts empty) and can be overridden from the stack environment.

**Migrations** (Alembic, from `backend/` with the venv active — URL comes from `DATABASE_URL`, no `-x` needed):

```bash
alembic upgrade head                 # apply pending migrations to the DB
alembic revision --autogenerate -m "…"  # after a model change, generate a migration
alembic downgrade -1                  # roll back one
```

On an **existing prod DB** whose pre-tenancy tables were built by `create_all()` (e.g. Neon), stamp the baseline as already-applied first, then upgrade — this applies the tenancy migrations (`0002`+`0003`), it does NOT recreate tables:

```bash
alembic stamp 0001 && alembic upgrade head
```

`0003` adds per-clinic unique constraints; if the existing data has, within one clinic, duplicate emails/phones or duplicate department names, dedupe before upgrading or the constraint creation fails.

Fresh dev SQLite needs no Alembic — `create_all()` on boot/seed builds the full current schema (incl. `clinic_id`). SQLite ALTERs run in Alembic batch mode (`render_as_batch`, set in `env.py`).

**Tests:** none in the repo. Verify behavior by driving the API (`/docs`, or curl against a running instance / container).

## Backend architecture

- **Config is centralized.** `app/core/config.py` is the ONLY module that reads env vars — everything imports the cached `settings` object. Swapping `DATABASE_URL` moves SQLite→Postgres with zero code changes (`app/core/database.py` sets SQLite-only connect args conditionally). In `ENVIRONMENT=production`, `settings` refuses to boot with the default `SECRET_KEY` (`model_post_init`).
- **Schema: `create_all()` for fresh DBs, Alembic for existing ones.** Fresh databases are still built by `Base.metadata.create_all()` (in `app/main.py` lifespan and `app/seed.py`) — that path always reflects the current models, including `clinic_id`. But `create_all()` never ALTERs an existing table, so schema changes to a live DB (e.g. Neon prod) go through **Alembic** (`backend/alembic/`, config in `alembic.ini`, URL pulled from `settings.DATABASE_URL` by `alembic/env.py`). Migration chain: `0001` = pre-tenancy baseline (a stamp target for DBs that already have those tables), `0002` = adds `clinics` + nullable `clinic_id` + backfills to a `default` clinic, `0003` = flips `clinic_id` to NOT NULL (all tenant tables except `users`) + per-clinic uniqueness, `0004` = adds `clinics.theme_preset` + `clinics.theme_overrides` (Session 3 theming; NOT NULL with server defaults, ALTER-only on existing DBs), `0005` = adds the two Session 4 tables `clinic_sections` + `media_assets` (pure additive `create_table`, no existing table touched). New model files MUST be imported in `app/models/__init__.py` or they won't register on `Base.metadata` (and Alembic autogenerate won't see them).
- **Multi-tenant (shared-schema row-level tenancy) — DONE (Session 2).** One API + one DB serve many clinics, fully isolated. `Clinic` (`app/models/clinic.py`, `slug` unique + `custom_domain`) is the tenant; every tenant-owned table (`Department`, `Doctor`, `DoctorAvailability`, `DoctorTimeOff`, `Appointment`, `ApiKey`, `VerificationCode`) carries a NOT NULL `clinic_id` FK. **`users.clinic_id` is nullable on purpose** — a `SUPERADMIN` is a global developer account with `clinic_id=NULL`; every PATIENT/ADMIN row has it set. Uniqueness is **per clinic**: `users(clinic_id,email)` / `(clinic_id,phone)` and `departments(clinic_id,name_ar|name_tr)` (so the same email/dept name can exist in different clinics); `api_keys.hashed_key` stays globally unique. Resolution deps (`app/dependencies.py`): `get_current_clinic` (X-Clinic slug header, fallback host/Origin → `custom_domain`; 400 if unresolved, 403 if inactive) for web/header endpoints, `get_current_clinic_optional` (login only — lets a headerless superadmin through), `get_api_key_clinic` (tenant = the clinic owning the API key, headers ignored) for `/api/v1/public/*`. **Every router query filters by the resolved clinic** — public reads scope by the X-Clinic header, JWT endpoints by `current_user.clinic_id` / `current_admin.clinic_id`, and `/public/*` by the key's clinic. `services.py` is unchanged: it's only ever called after the endpoint has verified the `doctor_id` belongs to the resolved clinic, and every row it reads is keyed to that doctor. **The cross-tenant leak is the failure mode that matters** — any new query on a tenant table MUST filter by `clinic_id`.
- **Superadmin (the developer).** `UserRole.SUPERADMIN`, guarded by `get_current_superadmin`. `/api/v1/superadmin/*` (`routers/superadmin.py`) is the ONLY place clinics are created/listed/deactivated and a clinic's first admin is created. Seeded from `SUPERADMIN_*` env vars with `clinic_id=NULL`. It logs in via `/auth/login` with **no** X-Clinic header (the optional-clinic dep falls back to a global superadmin lookup); it is NOT a clinic-admin, so it gets 403 from clinic-scoped endpoints.
- **Per-clinic theming (Session 3) — DONE.** Two layers merge into each clinic's look. (1) A developer-controlled **preset** (`backend/app/themes/*.json`, e.g. `default.json`, `modern.json`) — the big decisions: font family/imports, full colour palette, border radius, layout density. Only the developer edits these files; only the superadmin switches a clinic's preset (`clinic.theme_preset`, via `PATCH /superadmin/clinics/{id}/preset`). (2) The admin-editable **overrides** layer (`clinic.theme_overrides`, a JSON column) — a small safe subset: the primary/secondary/accent colours, logo URL, and display name + hero/contact/footer texts (per language). `app/theme.py:effective_theme(clinic)` merges preset + overrides; the preset is authoritative for everything the admin can't touch, so a broken/empty overrides blob still yields a valid theme. **`GET /api/v1/public/theme` is NO-auth**, tenant resolved by the X-Clinic header — it lives in `routers/theme.py:public_router` (a separate router) because `routers/public.py` is API-key-guarded. Admin reads/writes go through `routers/theme.py:admin_router` (`GET`/`PUT /api/v1/admin/theme`, scoped to `admin.clinic_id`). Overrides input is validated (hex colours, constrained logo URL) so theme data can't smuggle CSS/markup. PUT semantics = full replace of the overrides layer; the admin panel always resubmits the complete form state.
- **Media library + homepage sections (Session 4) — DONE.** Images live in **Cloudflare R2** (S3-compatible); the DB holds only tenant-scoped handles. Two tenant-owned tables (`app/models/media.py`, `app/models/section.py`), both NOT NULL `clinic_id`: `MediaAsset` (kind = logo/doctor_photo/equipment/gallery/section_image, `object_key`, `url`, alt texts, nullable `section_id`) and `ClinicSection` (kind = gallery/equipment/team/custom, bilingual title+body, `sort_order`, `is_active`). **Upload is presigned-PUT, never through the API:** `POST /admin/media/presign` validates content-type (`app/storage.py:ALLOWED_IMAGE_TYPES`) + size (`MEDIA_MAX_BYTES`) and mints a **server-generated, clinic-prefixed** `object_key` (`clinics/{clinic_id}/{uuid}.ext`) — the browser PUTs bytes straight to R2, then `POST /admin/media` persists the row. **The clinic prefix on the key is the tenant isolation** — `confirm`/`delete` reject any key not under `clinics/{admin.clinic_id}/` (`storage.belongs_to_clinic`), so a key can only ever land in its own clinic's namespace. `app/storage.py` is the ONLY R2 module (lazy boto3, cached client at `https://{account}.r2.cloudflarestorage.com`); with R2 unconfigured `is_enabled()` is False and the media endpoints return 503 (rest of the app boots fine — boto3 import is lazy). Admin CRUD (`routers/media.py:admin_router`, scoped to `admin.clinic_id`): media list/patch/delete (R2 delete is best-effort — DB row is source of truth), section create/patch/delete/**reorder** (`POST /admin/sections/reorder` takes the full ordered id list). Deleting a section **detaches** its images (`section_id → NULL`, `ondelete=SET NULL`) so R2 objects survive. Doctor photos and the theme logo are just a `MediaAsset.url` written onto `doctor.photo_url` / `theme_overrides.logo_url` via their existing endpoints — no new columns. **`GET /api/v1/public/sections` is NO-auth** (X-Clinic tenant, active sections + images), in `routers/media.py:public_router` for the same reason as `/public/theme`. `boto3` is now in `requirements.txt`.
- **Slots are computed, never stored.** `app/services.py:get_slots_for_doctor_on_date` generates bookable slots on the fly from a doctor's recurring weekly `DoctorAvailability` templates, minus `DoctorTimeOff`, minus non-cancelled `Appointment`s, minus past times. Editing a doctor's hours instantly affects all future dates. Booking calls `find_matching_slot` to re-validate the requested time server-side. Time arithmetic uses a fixed anchor date so a slot window crossing midnight terminates instead of looping.
- **Two auth models, both in `app/dependencies.py`:**
  - JWT Bearer (`get_current_user` / `get_current_admin` / `get_current_superadmin`) for the web portal. The role is always re-read from the DB row — the token's role claim is never trusted for authorization. Admin endpoints take the admin object as a param and scope every query to `admin.clinic_id`. `weekday` convention is Python's `Monday=0 … Sunday=6`.
  - API keys (`get_api_key`, `X-API-Key` header) for `/api/v1/public/*` — server-to-server bot/HIS integrations that book by phone number with no login. Keys are stored SHA-256-hashed; the raw key is shown once at creation. The key's `clinic_id` (via `get_api_key_clinic`) is the tenant for every `/public/*` query.
- **OTP flow** (`app/routers/auth.py`): password-reset ONLY — there is no signup verification. Registration creates the account immediately active (`is_verified=True`) and returns a JWT; there are no `/auth/verify/*` endpoints and no `verify.html` page. `VerificationCode` codes are bcrypt-hashed, expire, and are capped at `MAX_OTP_ATTEMPTS` wrong guesses via the shared `_consume_code` helper. `VerificationPurpose.ACCOUNT_VERIFY` still exists in the enum only so old DB rows deserialize — never issue it. Delivery is pluggable (`app/notifications.py`): `console` (logs the code — the zero-setup default) vs `smtp`/`twilio`, selected purely by env.
- **Concurrency:** booking endpoints lock the doctor row with `with_for_update()` to serialize concurrent bookings (real lock on Postgres/MySQL, no-op on SQLite). In `public.py`, patient find-or-create uses `flush()` not `commit()` so the lock is held until the appointment commits atomically.
- **One concern per file** under `models/`, `schemas/`, `routers/`. Emails are normalized to lowercase on register and on every lookup. Deleting a department with doctors, or a doctor with appointments, is rejected (409) to avoid FK orphans — deactivate (`is_active=False`) instead.

## Frontend architecture

- **Global objects, load order matters.** Scripts are plain `<script>` tags (no modules). Each page loads `config.js → auth.js → api.js → theme.js → i18n.js → layout.js → [icons.js] → <page>.js`. These define globals: `CLINIC_CONFIG`, `Auth`, `Api`, `Theme`, `I18n`, `Layout`, `Icons`. Page logic is one IIFE per file (`home.js`, `booking.js`, `admin.js`, `my-appointments.js`) or inline in the auth HTML pages.
- **`config.js` is the only place the API URL lives** (`API_BASE_URL`) — edit it when deploying the backend elsewhere. Also holds `CLINIC_SLUG` (the tenant — one deployed frontend = one clinic; `api.js` sends it as the `X-Clinic` header on every request, so it must match a backend clinic `slug`) and clinic name/logo.
- **XSS: escape all API/user data going into `innerHTML`.** The codebase builds DOM via template-literal `innerHTML`. Every interpolation of server/user text MUST be wrapped in `esc()` (defined in `api.js`) — patient names, doctor names, department names, notes, key names, etc. `textContent` assignments and numeric IDs don't need it. i18n strings come from trusted static JSON.
- **Theming:** `Theme` (`theme.js`, loaded right after `api.js`) fetches `GET /public/theme` and applies it as CSS custom properties on `:root` (`--color-*`, `--radius-*`, `--font-*`), so the same deployed frontend paints a different brand per clinic. Anti-FOUC: a cached copy (localStorage, keyed by `CLINIC_SLUG`) is applied synchronously the instant the script runs, then a fresh copy is fetched and re-applied — theming never blocks a page, and a down backend falls back to cache/`config.js` defaults. The theme also feeds display name + logo back into `CLINIC_CONFIG` (so `I18n.clinicName()` and `Layout` pick them up) and drives `[data-theme-text="path"]` nodes (e.g. `hero.title`, `hero.subtitle`, `footer`, `contact.phone`) per active language via `textContent` (no XSS). It fires `clinic:themechange`; `Layout` listens to swap the logo/brand. The stylesheet was already fully CSS-variable-driven, so theming just overrides those variables. The admin **Theme** panel (`admin.js`) edits the overrides layer with colour pickers (live-preview by setting the `--color-*` var), per-language text inputs, and a logo URL; it only persists colours the admin actually changed (untouched colours stay preset-driven). The Theme panel's logo field also has an **Upload** button that pushes a file to R2 (`Api.uploadMedia`, kind `logo`) and drops the returned URL into the field — a normal theme save then persists it.
- **Media & Homepage (Session 4):** `Api.uploadMedia(file, kind, meta)` (`api.js`) does the three-hop R2 upload — presign → **bare** `fetch` PUT to R2 (only the `Content-Type` header the presign signed; no Authorization/X-Clinic or the signature breaks) → confirm — and returns the `MediaAsset`. The admin **Media & Homepage** panel (`admin.js`, `loadMedia`) has a media library (upload, per-image assign-to-section / assign-to-doctor selects, delete) and homepage-section CRUD (add/edit/delete, ↑/↓ reorder via `/admin/sections/reorder`, upload-into-section, detach). The public homepage (`home.js:renderSections` into `#clinic-sections` in `index.html`) fetches `GET /public/sections` and renders each active section (bilingual title/body per active language, image grid, `loading="lazy"`), re-rendering on `clinic:langchange`; **every** server string goes through `esc()` (image `url`/alt into `src`/`alt` included) and a missing/failed endpoint just renders nothing.
- **i18n:** `I18n` (`i18n.js`) loads `assets/i18n/{ar,tr}.json`, sets `<html lang dir>` for RTL/LTR, translates `[data-i18n]` / `[data-i18n-placeholder]`, and fires a `clinic:langchange` event that pages listen to in order to re-render API-driven content in the new language. Add a language by copying a JSON file and adding a `.lang-switch` button.
- **Session:** `Auth` keeps JWT + user in `localStorage`. `Auth.requireAuth()` / `requireAdmin()` guard pages client-side (real enforcement is server-side). The `?next=` redirect after login is restricted to same-site `*.html` names.
- **Booking wizard** (`booking.js`): 4 steps (department → doctor → date/time → confirm). If a user hits "confirm" while logged out, the selection is stashed in `localStorage` (`clinic_booking_draft`) and restored after the login redirect.

## Bot service (Session 5) — DONE

Standalone, channel-agnostic AI appointment assistant living in its own top-level **`bot/`** directory — a separate FastAPI app with its own `requirements.txt`, `Dockerfile`, and `.env`. **Hard rule: it never imports `backend/app` and never touches the DB.** Its ONLY access to clinic data is the public REST API (`/api/v1/public/*`) authenticated with `X-API-Key` — exactly how the future WhatsApp/Telegram bots connect.

- **Channel-agnostic core.** `bot/app/engine.py:ConversationEngine.handle(channel, conversation_id, message, init_identity?)` returns reply text; transport lives in `bot/app/adapters/` (`base.py` = `ChannelAdapter` ABC + normalized `InboundMessage`; `web.py` = the implemented `web` adapter). Adding WhatsApp/Telegram = a new adapter, no engine change. The web adapter is served by `POST /bot/v1/chat` (JSON in/out); the client holds a `conversation_id` and sends it every turn.
- **Tenant resolution.** `bot/app/config.py:BOT_TENANTS` is a `tenant=apikey` registry (the ONLY module reading env). The client sends the tenant as the **`X-Bot-Tenant`** header (one deployed widget = one clinic, mirroring the frontend's `X-Clinic`); unknown tenant → 404. Each tenant's clinic API key is created in the admin panel. `bot/app/clinic_api.py:ClinicApiClient` is the thin async httpx wrapper over the public API — every call carries that key, so the backend resolves the clinic from the key itself.
- **LLM: Gemini 1.5 Flash** via `google-generativeai` (manual function calling; the package is deprecated upstream but pinned intentionally per the roadmap). `GEMINI_API_KEY` from env; with it blank `/bot/v1/chat` returns 503, `/health` reports `gemini:false`. Tools (`bot/app/tools.py`, `genai.protos` FunctionDeclarations + async executors): `provide_identity`, `list_departments`, `list_doctors`, `get_available_slots`, `book_appointment`, `list_my_appointments`, `cancel_appointment`, `reschedule_appointment` — each a thin wrapper over one public-API call. Engine loop caps tool iterations (`MAX_TOOL_ITERATIONS`).
- **Identity & isolation — enforced in code, never trusted to the model.** The patient is identified by phone, stored in conversation state (`bot/app/state.py`, in-memory `(tenant, conversation_id)` store with TTL eviction; swap for Redis/SQLite behind the same interface for multi-replica). **Tools that read/modify appointments take NO phone argument in their schema** — the executor injects the state's phone (`book`/`list`/`cancel`/`reschedule`), so the model literally cannot request another phone's data. A logged-in web client may pass a trusted `identity` at init (set once, never overwritten, never from the model); otherwise the bot collects name+phone conversationally and records it via `provide_identity`.
- **Safety rails outside the prompt** (`bot/app/safety.py` + engine/state): max message length (`MAX_MESSAGE_CHARS` → 413), per-conversation sliding-window rate limit (`RATE_LIMIT_PER_MINUTE`), recursive prompt-injection stripping of tool-result strings (clinic free-text like doctor bios/notes can't hijack the model), history trimming (`MAX_HISTORY_TURNS`), and per-tool logging (never cross-tenant). System prompt (`bot/app/prompt.py`): clinic-only scope, 1–3 sentence replies, mirrors patient language (Arabic/Turkish), never invents medical advice or availability.
- **Backend addition for Session 5:** the only backend change is `POST /api/v1/public/appointments/{id}/reschedule` (API-key auth, clinic-scoped) — booking/list-by-phone/cancel already existed. It locks the doctor row, flushes the appointment to `CANCELLED` so the old slot frees, re-validates the new slot via `find_matching_slot`, then reassigns the same row (rollback on conflict/out-of-hours).
- **Provisioning a clinic's bot:** admin creates an API key → developer adds a `tenant=key` pair to `BOT_TENANTS` → point `CLINIC_API_BASE_URL` at the backend and set `CORS_ORIGINS` to the widget origin. Compose service `clinic-bot` (`backend/docker-compose.yml`, built from `../bot`, port `9001:9000`) reaches the API over the compose network (`http://clinic-api:8000/api/v1`).

## Deployment notes

- Frontend is hosted on **Cloudflare Pages** (build output dir = `frontend`, no build command).
- Backend runs on a **VPS via Portainer** (stack from this repo, compose path `backend/docker-compose.yml`), exposed over HTTPS through a **Cloudflare Tunnel** (`cloudflared` service → `clinic-api:8000`; host port mapping `8001:8000` is only for direct VPS access). Because frontend is HTTPS, the API must be HTTPS too (mixed-content otherwise).
- Current production hostnames: API at `https://clinic-api.dijivoo.com` (what `config.js` points to), frontend at `https://clinic.dijivoo.com` (allowed in `CORS_ORIGINS`). Data lives in **Neon** (managed Postgres), not on the VPS.
- After changing where either half is hosted: update `API_BASE_URL` in `config.js` AND `CORS_ORIGINS` (exact origin, no trailing slash) on the backend — a mismatch silently blocks every API call in the browser.
- **Cloudflare R2 (media, Session 4):** create an R2 bucket (Cloudflare dashboard → R2), then an R2 **API token** (R2 → Manage API Tokens → Object Read & Write) — its Access Key ID/Secret go in `R2_ACCESS_KEY_ID`/`R2_SECRET_ACCESS_KEY`, and `R2_ACCOUNT_ID` is your Cloudflare account id (the S3 endpoint `https://{account}.r2.cloudflarestorage.com` is built from it). Give the bucket **public read**: enable the `r2.dev` dev URL or bind a custom domain, and put that base (no trailing slash) in `R2_PUBLIC_BASE_URL` — object URLs are `R2_PUBLIC_BASE_URL/{object_key}`. Two CORS rules on the bucket: allow `PUT` from the frontend origin (browser presigned upload) and `GET` from anywhere (public image reads). Leave the vars blank to run without media (uploads then 503). These are passed through `backend/docker-compose.yml` from the stack environment.
- **Bot service (Session 5):** compose service `clinic-bot` (built from `../bot`). Required stack env: `GEMINI_API_KEY` (Google AI Studio), `BOT_TENANTS` (`tenant=apikey,…`). Optional: `GEMINI_MODEL`, `BOT_CLINIC_API_BASE_URL` (defaults to the in-network `http://clinic-api:8000/api/v1`), `BOT_CORS_ORIGINS` (widget origin), `BOT_MAX_MESSAGE_CHARS`/`BOT_RATE_LIMIT_PER_MINUTE`/`BOT_CONVERSATION_TTL_SECONDS`. Expose `/bot/v1/chat` over the tunnel like the API when the widget lands (Session 6).
- `git push` uses HTTPS (SSH may be blocked on some networks); auth needs a GitHub PAT.
