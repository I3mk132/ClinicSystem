# Clinic Booking System — Backend (FastAPI)

REST API only — no server-rendered pages. The `frontend/` folder (or any
future mobile app) talks to this API over HTTP/JSON.

## Setup

```bash
cd backend
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

pip install -r requirements.txt

cp .env.example .env             # then edit SECRET_KEY, admin email/password, etc.

python -m app.seed               # creates tables + default admin + demo data

uvicorn app.main:app --reload    # http://127.0.0.1:8000
```

Interactive API docs (auto-generated by FastAPI): **http://127.0.0.1:8000/docs**

Default admin login (change it in `.env` before going live):
```
email:    admin@myclinic.com
password: Admin@12345
```

## Project layout

```
backend/
  app/
    core/
      config.py       # ALL environment configuration lives here
      database.py     # SQLAlchemy engine/session (swap DB via .env only)
      security.py     # password hashing, JWT, OTP codes, API keys
    models/           # SQLAlchemy tables (one concern per file)
    schemas/          # Pydantic request/response shapes
    routers/          # one FastAPI router per resource (auth, doctors, public, ...)
    dependencies.py   # get_current_user / get_current_admin / get_api_key guards
    services.py       # slot-generation business logic (booking engine)
    notifications.py  # pluggable email/SMS sending for OTP codes
    seed.py            # creates default admin + demo data + demo API key
    main.py            # FastAPI app, CORS, router registration
  requirements.txt
  .env.example
```

This mirrors a typical "clean architecture" split so you can find things
fast: **models** = what's stored, **schemas** = what the API accepts/returns,
**routers** = HTTP endpoints, **services** = business rules that don't
belong to any single endpoint (like slot generation).

## How booking availability works

Nothing about "free time slots" is stored directly. Instead:

1. Admin defines a **weekly template** per doctor (`DoctorAvailability`):
   e.g. "Sundays 09:00–13:00, 30-minute slots".
2. Admin can add one-off **time off** (`DoctorTimeOff`) for holidays etc.
3. When the frontend asks
   `GET /api/v1/doctors/{id}/available-slots?date=2026-07-12`,
   the backend generates that day's slots from the template, removes
   anything in `DoctorTimeOff`, and marks slots already taken by a
   non-cancelled `Appointment` as unavailable.

Editing a doctor's weekly hours instantly applies to every future date —
no migration or slot table to maintain.

## Switching the database (SQLite → PostgreSQL/MySQL)

Everything goes through SQLAlchemy, so scaling up is a one-line change in
`.env`:

```
# Development (zero setup)
DATABASE_URL=sqlite:///./clinic.db

# Production
DATABASE_URL=postgresql+psycopg2://user:password@host:5432/clinic_db
```

Then install the matching driver (see the commented lines in
`requirements.txt`) and re-run `python -m app.seed`. No application code
changes needed.

> For a production deployment, replace the `Base.metadata.create_all()`
> call in `main.py`'s startup event with proper migrations (e.g. Alembic)
> so schema changes are versioned.

## Adding a new field / resource

1. Add the column to the relevant model in `app/models/`.
2. Add/adjust the matching Pydantic schema in `app/schemas/`.
3. Expose it through the router in `app/routers/`.
4. Delete `clinic.db` (dev only) or write a migration, then re-seed.

## Authentication

- JWT bearer tokens (`Authorization: Bearer <token>`), issued by
  `POST /auth/login` and `POST /auth/register`.
- Two roles: `patient` (self-registered) and `admin` (created via
  `app/seed.py` or promoted directly in the database).
- `app/dependencies.py` exposes `get_current_user` and `get_current_admin`
  as FastAPI dependencies — add them to any router that needs protecting.
- Login accepts **either an email or a phone number** as the `identifier`
  field, whichever the person signed up with.

## Account verification & forgot password

At signup, the person picks a `contact_method` (`email` or `phone`) —
that's where their one-time codes (OTP) get sent, regardless of which
identifier they log in with:

- `POST /auth/verify/request` / `POST /auth/verify/confirm` — confirm a
  new account (sent automatically right after registration; the frontend
  also offers a "resend" button on `verify.html`).
- `POST /auth/password/forgot` (`{ "identifier": "..." }`) / `POST /auth/password/reset`
  — the classic "forgot password" flow, code delivered to the account's
  `contact_method`. `forgot` always returns the same generic message
  whether or not an account exists, to avoid leaking which emails/phones
  are registered.

Codes are 6-digit, expire after `OTP_EXPIRE_MINUTES` (default 10), are
single-use, and stored hashed (`app/models/verification.py`).

### Where the codes actually get sent: `app/notifications.py`

Two independent, swappable channels, selected purely from `.env` — **no
code changes needed** to go from local development to production:

| Setting | Values | What it does |
|---|---|---|
| `EMAIL_PROVIDER` | `console` (default) / `smtp` | `console` just logs the email to the backend terminal — zero setup, perfect for development. `smtp` sends a real email via any SMTP server (Gmail, SendGrid, your own mail server — see `.env.example` for examples). |
| `SMS_PROVIDER` | `console` (default) / `twilio` | Same idea; `twilio` sends a real SMS (`pip install twilio` first). |

Because it's abstracted behind `send_otp()`, adding another provider (e.g.
AWS SES, Vonage) later is a matter of adding one function in
`notifications.py` and one `if` branch — nothing else changes.

## Public API for integrations (bots, other systems)

Everything under `/api/v1/public/...` is designed for **server-to-server**
use — the main hospital/clinic system, a WhatsApp bot, a Telegram bot,
etc. — authenticated with a simple API key instead of a user login:

```
GET  /api/v1/public/departments
GET  /api/v1/public/doctors?department_id=
GET  /api/v1/public/doctors/{id}/available-slots?date=
POST /api/v1/public/appointments        # book by name + phone number, no login needed
GET  /api/v1/public/appointments?phone= # list a patient's bookings by phone
POST /api/v1/public/appointments/{id}/cancel?phone=
```

Every request needs an `X-API-Key: <key>` header. Booking via this API
finds-or-creates a lightweight patient record by phone number (a bot
already knows who it's talking to from the chat platform, so no
password/login step is needed on that side).

**Managing keys:** Admin → **Integrations** tab in the web portal
(`admin.html`), or directly via `POST/GET/PATCH /api/v1/api-keys`
(admin-only, JWT-protected). The raw key is shown exactly once, at
creation — store it in your bot/integration's config right away. A demo
key is created automatically by `app/seed.py` and printed to the console
(and saved to `backend/.demo_api_key.txt`, gitignored) so you can try the
public API immediately.

Full interactive documentation for every endpoint (including these) is
always at **`/docs`**.

## Deploying with Docker (VPS + Portainer)

The backend ships with a production-ready `Dockerfile` + `docker-compose.yml`.
The container starts, seeds the database (tables + admin account, idempotent),
then serves the API on port `8000`. The SQLite database lives on a named
volume (`clinic_data`), so it survives container rebuilds/updates.

### Option A — Portainer stack from the Git repository (recommended)

1. Portainer → **Stacks** → **Add stack** → **Repository**
2. Repository URL: `https://github.com/I3mk132/ClinicSystem`
   Compose path: `backend/docker-compose.yml`
3. Add the **environment variables** (at minimum):

   | Variable | Value |
   |---|---|
   | `SECRET_KEY` | output of `python -c "import secrets; print(secrets.token_hex(32))"` |
   | `FIRST_ADMIN_PASSWORD` | a strong admin password |
   | `FIRST_ADMIN_EMAIL` | your admin email |
   | `CORS_ORIGINS` | where the frontend is served, e.g. `https://clinic.example.com` |
   | `TZ` | clinic timezone, e.g. `Europe/Istanbul` |

4. **Deploy the stack.** API is now on `http://<vps-ip>:8000` (docs at `/docs`).

### Option B — Upload the folder and use the shell

```bash
# on the VPS, inside the uploaded backend/ folder:
SECRET_KEY=... FIRST_ADMIN_PASSWORD=... docker compose up -d --build
```
(or put those variables in a `.env` file next to `docker-compose.yml` —
Docker Compose reads it automatically for `${...}` substitution.)

### After deploying

- **Point the frontend at the API**: set `API_BASE_URL` in
  `frontend/assets/js/config.js` to `https://<your-api-domain>/api/v1`.
- **HTTPS**: put a reverse proxy (Nginx Proxy Manager, Traefik, Caddy) in
  front of port 8000. The container already runs uvicorn with
  `--proxy-headers`, so client IPs/scheme are preserved.
- **CORS**: `CORS_ORIGINS` must exactly match the frontend origin
  (scheme + domain, no trailing slash) or browsers will block API calls.
- **OTP codes**: with the default `EMAIL_PROVIDER=console`, verification
  codes are printed to the container logs (Portainer → container → **Logs**).
  Switch to real SMTP by setting `EMAIL_PROVIDER=smtp` + the `SMTP_*` vars.
- **Safety rails**: with `ENVIRONMENT=production` (the compose default) the
  app refuses to boot with the default `SECRET_KEY`, and demo data seeding
  is disabled (`SEED_DEMO_DATA=false`), so a fresh deploy starts clean.
- **Upgrades**: redeploy the stack (or `docker compose up -d --build`);
  the `clinic_data` volume keeps the database.
- **Backups**: the whole database is one file — copy it out with
  `docker cp clinic-api:/data/clinic.db ./clinic-backup-$(date +%F).db`.
- **PostgreSQL later**: uncomment the `clinic-db` service + Postgres
  `DATABASE_URL` in `docker-compose.yml`. No code changes required.
